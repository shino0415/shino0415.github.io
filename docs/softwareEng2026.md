# ソフトウェア工学まとめ
第1回から第6回までのまとめ
## ソフトウェア工学とは
### ソフトウェアの定義
- 実行されることによって必要な特性、機能、性能を提供する命令語群
- プログラムが適切に情報を扱うことを可能とするデータ構造
- プログラムの操作や使用法記述した情報

### ソフトウェア工学の目指す先
**ソフトウェアシステムが変化し続ける状況に対応する方法論を考え出すこと**  
ソフトウェアは変化する。そのため、変化することを前提につくらなければならない。変化させやすさの視点や、変化した際も互換性を保つことができるかの視点が必要となる。

### Hookerの7原則
1. システムは、ユーザーに価値を提供するためにある
2. シンプルであるべき
3. ビジョンをもつべき
4. 利用するのは作成者ではない
5. 未来に選択肢を多く残すべし
6. コスト低減、再利用するコンポーネントとそれを取り込むシステム両方の価値を高める
7. 行動を起こす前に考えを持つべし

### ソフトウェア工学の必要性
1. ソフトウェア利用の**大規模化**  
ソフトウェアがインフラ化したため、ソフトウェアを効率よく開発・メンテナンスする必要がある
2. **IT人材の不足**  
ソフトウェアの需要に対するＩＴ人材が追いついていない  
また、ソフトウェアを効率的に開発・メンテナンスすることができる人材が少ない
3. **不測の事態に対応した**事業・サービスの継続が求められている
4. **不確実性の増大**  
ソフトウェア利用が変化したり、多様化したりしている。それにより製作者の意図しない目的に活用されることもある

## ソフトウェアライフサイクル
### 大まかなサイクル
1. 計画
2. 要件定義
3. 設計
4. 製作
5. テスト
6. 運用・保守
### 計画
**ニーズを想定する**
- ユーザ
- 顧客
- システム
- ベンダ
### 要件定義
**実現する/しない機能を明確にする**
以下を定義する
- システムの目的
- システムの概要
- システムの機能
- システムの構成
- 目標性能
- 他システムとのインタフェース仕様
- 運用面の注意事項
- 制限事項
- 拡張性
- 開発スケジュール
- 開発体制
- 納品物

要件定義を行う意義
- プロジェクト関係者内でのスコープの共有
- 社内外への宣言や協力依頼がしやすくなる
- RFP
### 設計
**要件定義をもとに設計書を作る**  
- コーディングができるレベルの仕様書に
- WBSの作成
### 製作
**設計書をもとにシステム構築を行う**
- 内製（自社開発）
- 調達（外部委託）
    - オフショア開発  
    人件費の安い海外企業に外部委託する手法
### テストとデバッグ
**あらかじめテストしバグを早期に発見し、対処する**
- バグのないソフトウェアは存在しない
- 全てをテストすることは不可能
### 保守・運用
- 稼働後のデータをプロアクティブに活用するべき
- ソフトウェア以外の要因も考慮する必要がある
- 開発者はプロジェクトから離れている可能性がある
### ドキュメンテーション
ソフトウェア開発・運用において、ドキュメンテーションは大切となる。
- 複数人でのソフトウェア開発
- 非同期コミュニケーション
- 日本語とは限らない

## 読みやすいコード
コードは書くよりも読まれることの方が多い  
読みやすいコードを書くことは重要である  
### 読みやすいコードとは
- 1行の長さは79文字以内
- インデントするごとにスペースを4つ使う（Tabではなく）
- 演算子の前後にスペースを1つずつ空ける
- 不要なスペースを入れない
- 1つの行には1つの指示のみ
- 計算式の途中で改行する場合は、演算子の位置を揃える
- コードの内容と矛盾しないコメント
- 基本的にコメントは英語で記載する
- 変数や定数の命名規則に忠実に命名する

## git
分散管理型のバージョン管理システム
### コミット
- ファイル作成/変更/削除の記録
- コミットメッセージ（どのような変更をくわえたか）
### レポジトリ
- gitが管理するプロジェクトのフォルダ
### ブランチ
- 作業を枝分かれさせることができる
- ブランチにより、共同作業・並行作業が可能になる

## CI/CD
### CI
- コード変更を共有リポジトリに頻繁に統合するプロセス
- 自動テストとビルドを定期的に実行し、バグの早期発見と修正を可能にする
### CD
- コード変更をテスト環境や本番環境に自動的にデプロイするプロセス
- ユーザーフィードバックを迅速に反映可能
### 基本的なステップ
1. **ソース**  
コード変更をトリガーにワークフローを起動
2. **ビルド**  
ソースコードをコンパイル
3. **テスト**
4. **デプロイ**
5. **検証**  
デプロイされたアプリケーションの動作確認
6. **モニタリング**  
本番環境での継続的な監視
